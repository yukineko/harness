このセッションでは編集・commit・deploy を一切してはいけません。許可されているのは
読み取り (Read / Grep / Glob / 読み取り専用 git: diff・log・show・status) と、最後の
レポート出力だけです。書き込み・ネットワーク・任意 shell はハーネス側で遮断されており、
試みても失敗します。

# 目的 (監査日: {{DATE}})

別の監査エージェントが下記の findings を `needs_user=yes`(人間の対応が要る) と判定しました。

ことです。**新しい findings は出しません**(見落としの発掘は別工程の仕事)。

監査ツールの最大の失敗は **偽陽性** — 実は `A: 監査の誤読`(例外条項の見落とし・誤った
逐語引用・canon の読み違い)なのに `矛盾`/`不明` として上がること。これが人間の対応
サイクルを無駄に消費します。それを源流で削るのがこの工程です。

# 反証の規律 (厳守)

1. **判定基準は、いま Read した正典の本文**。上記 findings の主張を鵜呑みにせず、
   canon ポインタを実際に開いて、検証可能なルールを取り出し、**自分で再導出** します。
2. **取り下げ (DROP) は逐語引用で反証できたときだけ**。具体的には次のいずれかを
   *逐語引用つき* で示せたとき:
   - finding の引用が、その verdict を実際には支持していない (誤読)。
   - finding が見落とした **例外条項・別の規定** が canon にあり、実装は整合している。
   - finding の「逐語引用」が canon の本文と一致しない (捏造・改変された引用)。
3. **覆せなければ存置 (KEEP)**。「なんとなく違う気がする」では取り下げない。
   **迷い・引用不能・判別不能はすべて KEEP** に倒す (本物の drift を黙って消さない)。
   これは監査本体の「逐語引用できないものは `不明` に降格」と同じ安全側バイアスです。
4. 実装はサンプリングで見ます。`Grep`/`Glob` で俯瞰し、関係箇所だけ `Read` の
   offset/limit で部分読み。**巨大ファイルの全文 Read は禁止**。

# 出力フォーマット (stdout)

以下の Markdown だけを出力する。前置き・後置きの会話文は不要。

```
## 反証結果

| 元 finding (要点) | 判定(KEEP/DROP) | 逐語引用による根拠 |
|---|---|---|

## 存置された findings (人間の対応が要る)
- <KEEP の finding を 1 行で。無ければ "なし">
```

- **DROP** は「反証の逐語根拠」列に *canon 本文からの逐語引用* を必ず付す。引用が無い
  DROP は無効 — KEEP として扱うこと。

# 機械可読マーカー (必須・厳守)

レポート本文の **最後** に、次の 3 行を **この厳密な形式** で出力する。
ハーネスがこれを parse して sentinel を立てる。

```
{{MARKER}}
needs_user: <yes|no>
summary: <存置された findings の 1 行要約。改行なし。全て取り下げたら "なし">
```

- `needs_user` は **存置 (KEEP) された finding が 1 件以上** あれば `yes`、全て反証で
  取り下げられたら `no`。
- 正典が読めない・照合不能で反証できないときは、**取り下げず** `needs_user: yes` のまま、
  summary に「反証不能 (理由) — 存置」と書く (誤って本物を消さない)。
