# Plugin activation scopes

Every plugin in this monorepo is classified by **how loud its activation surface
is** — i.e. how often its code runs relative to the model's turns. This keeps the
always-on hook / injection budget (see
[`crates/ctxrot/docs/adr/0001-cross-harness-injection-budget.md`](../crates/ctxrot/docs/adr/0001-cross-harness-injection-budget.md))
legible: you can see at a glance which plugins pay a per-turn cost and which only
run on demand.

The taxonomy is generated, not hand-maintained. Regenerate the classification
below with:

```sh
cargo run -q -p harness-status -- plugins          # human table
cargo run -q -p harness-status -- plugins --json    # machine-readable + counts
```

## The three scopes

The scanner reads each crate's `hooks/hooks.json` (collecting the top-level event
keys), plus whether it has non-empty `skills/` and `agents/` dirs.

- **Always-on** — registers a hook on a **high-frequency lifecycle event** that
  fires basically every turn/session. The always-on event set is:
  `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `Stop`,
  `SubagentStop`. Any one of these makes a plugin always-on. These are the
  plugins whose cost is paid on (nearly) every turn, so they are the ones the
  injection/latency budgets watch.
- **Event-scoped** — registers hooks **only** on lower-frequency / conditional
  events (none of the always-on set): e.g. `PreCompact`, `SessionEnd`,
  `Notification`. They run, but rarely — not on the hot path of every turn.
- **Manual** — registers **no hooks at all**. Reachable only via a skill (slash
  command) or direct CLI. A Manual plugin with a skill is a slash-command; one
  with neither skill nor agent and only a binary (like `harness-status`) is
  CLI-only.

Classification rule (pure, unit-tested in `crates/harness-status/src/plugins.rs`):
if `hook_events` contains any always-on event → **Always-on**; else if
`hook_events` is non-empty → **Event-scoped**; else → **Manual**. `has_skills` /
`has_agents` never change the scope — they only inform the human "trigger"
description.

A crate dir is treated as a plugin iff it has any of `.claude-plugin/`, `hooks/`,
`skills/`, or `agents/`. Plain library crates (`harness-core`, `mutategate`) have
none of these and are **excluded** — they present no plugin surface.

## Why harness-status stays manual

`harness-status` is deliberately **Manual / CLI-only**. It would be trivial to add
a `SessionStart` dashboard hook so the status view auto-injects at the start of
every session — but that is exactly the wrong move. A `SessionStart` hook is an
always-on event: it would grow the always-on injection/hook budget that ADR 0001
and the `harness-status inject` / `harness-status hooks` monitors exist to *curb*.
Making the very tool that measures the budget also *spend* the budget would be
self-defeating.

So `harness-status` stays the **unified manual HOTL inspection dashboard**. It is
read-only, holds no hooks, and is run on demand — by a human, or via its
`/status` skill — when you actually want to look:

- `harness-status` — full dashboard (budget + sessions + progress + hook latency + inject)
- `harness-status budget` — today's spend
- `harness-status sessions` — recent sessions
- `harness-status progress` — current progress file
- `harness-status hooks` — Stop-hook latency aggregation (budget monitor)
- `harness-status inject` — UserPromptSubmit injection-size aggregation (budget monitor)
- `harness-status plugins` — this activation-scope classification

Its own activation scope is therefore **Manual**, on purpose, and it appears as
such in the table above.
