# 目的

下記の **要望** を、Agent が実装可能な単位の **詳細仕様 (Spec IR)** に正規化します。
具体的には、要望を **原子的な requirement** に分解し、各 requirement に **反証可能な
acceptance criteria** を与えます。

**重要 — 生成を目的化しない (HOTL):** 「何か出力する」こと自体が目的ではありません。
**厳格に仕様化できる根拠があるときだけ** draft を出します。根拠が無いものを推測で
でっち上げてはいけません (specguard が引用できないものを `不明` に降格するのと同じ規律)。

# 厳格化ゲート (rigor gate) — 各 requirement が満たすべき4条件

各 acceptance criterion について、以下を **すべて** 満たせるか判定します:

1. **G1 接地 (traceable)**: その acceptance を裏付ける根拠が canon に **逐語引用 (verbatim)**
   できる。引用できないものは acceptance にしてはいけない。
2. **G2 沈黙ゼロ (no silence)**: acceptance を固めるのに必要な決定点について canon が
   沈黙していない (値・閾値・分岐が canon で決まっている)。
3. **G3 矛盾ゼロ (no contradiction)**: 要望と canon、または canon 同士が食い違っていない。
4. **G4 反証可能 (falsifiable)**: その criterion が観測可能な pass/fail に落ちている
   (「速い」「使いやすい」のような検証不能な表現は不可)。

## 判定の規律 (厳守)

- 判定基準は **いま Read した正典の本文**。このプロンプト内の要約ではありません。
- canon ポインタを実際に開き、**逐語引用** で裏付けてから acceptance を書くこと。
- 巨大ファイルの全文 Read は禁止。`Grep`/`Glob` で俯瞰し、関係箇所だけ部分読みする。
- **G2 が落ちる (沈黙/接地不能)** → その requirement は厳格化できない → 不足として扱う。
- **G3 が落ちる (矛盾)** → 機械が「どちらが正か」を決めない → 人間に上げる。

# 出力 (どちらか一方)

## A. 全 requirement が G1–G4 を満たせる場合 → draft を出す

レポート本文に、`[[requirement]]` の TOML を **必ず ```toml フェンスで囲って** 出力します
([spec] ヘッダはハーネスが付けるので書かないこと)。フェンスの前に短い判定理由を書いて
構いませんが、**requirement 本体は必ずフェンス内に** 入れること (ハーネスはフェンス内を
抽出します)。各 requirement は次の形:

```toml
[[requirement]]
id = "R1"
statement = "<その要望を1文の検証可能な要求に落としたもの>"
acceptance = ["<反証可能な受け入れ条件1>", "<条件2>"]
canon = ["<この requirement を裏付ける canon ポインタ>"]
falsifiable = true
```

そのあと、本文の最後に **次の3行ちょうど** を出力します:

```
{{MARKER}}
rigor: pass
needs_user: no
summary: <1行で要約>
```

## B. 1つでも厳格化できない場合 → でっち上げず、人間に上げる

draft を出さず、レポート本文に **何が不足か / 何を Ask するか / どんな文書を追加して
ほしいか** を書きます。必要なら **追加してほしい文書のフォーマットや雛形まで** 出して
構いません (人間が埋める型を渡す)。各不足には、落ちたゲート (G1–G4) と、可能なら
canon の **逐語引用** (沈黙の場合は「どこにも無い」ことの根拠) を添えます。

そのあと、本文の最後に **次の3行ちょうど** を出力します:

```
{{MARKER}}
rigor: fail
needs_user: yes
summary: <1行で要約 — 何が足りないか>
```
